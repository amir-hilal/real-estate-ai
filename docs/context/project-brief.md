# Project Brief: AI Real Estate Agent

> **Audience:** Engineers joining the project or reviewing it for the first time.  
> **Purpose:** Describe the system in precise engineering language. No vagueness. No marketing.

---

## 1. What This System Does

This system accepts a free-text English description of a property and returns:

1. A validated set of structured property features extracted from that description
2. A predicted sale price generated by a trained regression model
3. A grounded, human-readable explanation of how that price was derived

The system is not a search engine. It does not query live listings. It predicts based on statistical patterns learned from historical housing data (Ames Housing dataset) and communicates that prediction in terms a non-technical user can understand.

---

## 2. The Two-Stage LLM + ML Chain

The pipeline has four distinct processing stages that execute in sequence:

### Stage 1 — Feature Extraction (LLM)

**Input:** Free-text property description from the user  
**Process:** A structured prompt is sent to an LLM (e.g., GPT-4o, Claude) with explicit instructions to extract property attributes as a JSON object matching the system's feature schema.  
**Output:** A populated (or partially populated) `PropertyFeatures` Pydantic model

**Key behaviors:**
- The LLM is instructed to return only fields it can confidently infer from the text
- Fields the LLM cannot determine are returned as `null` or omitted
- The Pydantic schema enforces type constraints and valid value ranges
- If any **required** features are `null` after extraction, the request does not proceed to Stage 2

**Example:**  
> Input: *"3-bedroom house in the northwest of Ames, built in 1995, with a two-car garage and a finished basement."*  
> Output: `{ "bedroom_abv_gr": 3, "year_built": 1995, "garage_cars": 2, "bsmt_qual": "Good", ... }`

---

### Stage 1.5 — Missing Feature Collection (UI fallback)

**Input:** Partially populated `PropertyFeatures` model  
**Process:** The API returns the partially filled features along with a list of missing required fields. The frontend displays input controls for those specific fields only.  
**Output:** A fully populated `PropertyFeatures` model after user fills the gaps

This stage is not an LLM call. It is a validation-driven UI interaction. The ML model cannot run until all required features are present and validated.

---

### Stage 2 — Price Prediction (ML Model)

**Input:** Fully validated `PropertyFeatures` model  
**Process:** Features are transformed using the same preprocessing pipeline that was applied during training. The serialized model runs inference.  
**Output:** A numeric price prediction in USD, plus any available confidence signal (e.g., prediction interval if supported)

**Key constraints:**
- The preprocessing pipeline must be exactly the same as training-time — no divergence
- Categorical encodings must be loaded from training artifacts, not recomputed at inference
- Unknown categories (not seen during training) must be handled explicitly, not silently dropped

---

### Stage 3 — Prediction Explanation (LLM)

**Input:** Predicted price + validated feature values + summary statistics from training data  
**Process:** A second structured prompt is sent to an LLM. The prompt provides context: the predicted price, the property's features (in human terms), and relevant statistics (e.g., median price for that neighborhood, average lot size, price per square foot).  
**Output:** A 2–4 paragraph explanation of why the model arrived at that price, written for a non-technical audience

**Key behaviors:**
- The explanation must be grounded in the data provided to the prompt — no hallucinated facts
- The explanation must mention the predicted price explicitly
- The explanation must reference at least two statistical comparisons from the training data
- The explanation must acknowledge major contributing features

---

## 3. User Flow (End-to-End)

```
1. User opens the UI or sends POST /predict request
2. User enters a free-text description of a property
3. System sends description to Stage 1 LLM
4. LLM returns structured JSON feature set
5. System validates JSON against PropertyFeatures schema
   a. If required fields are missing → return partial features + list of missing fields
      → User fills missing fields via UI controls
      → User resubmits complete feature set
   b. If all required fields present → proceed
6. System runs ML model on validated feature set
7. System constructs explanation prompt with: prediction, features, training statistics
8. Stage 2 LLM returns plain-English explanation
9. System returns final response: { features, prediction, explanation }
10. UI displays predicted price and explanation; optionally shows extracted features
```

---

## 4. Value of the System

**Educational value:**
- Demonstrates the full pipeline from natural language → structured data → ML → natural language
- Forces correct engineering discipline: validation, schema design, leakage prevention, prompt versioning
- Each stage is independently understandable and testable

**Product value:**
- Users without ML knowledge can get an estimated price for a described property
- The explanation makes predictions trustworthy and actionable, not just a number
- Missing-field collection ensures the model always receives quality input, not garbage

## 5. Required Deliverables (MVP)

| Deliverable | Description |
|-------------|-------------|
| `PropertyFeatures` schema | Pydantic model with all ML-required features, types, and constraints |
| Stage 1 prompt | Versioned extraction prompt with clear output format instructions |
| ML training notebook | Documented EDA → preprocessing → training → evaluation notebook |
| Serialized model artifact | Trained model + preprocessing pipeline saved as a file |
| Training statistics file | Summary stats (mean, median, std, percentiles) for key features — used in Stage 2 |
| Stage 2 prompt | Versioned explanation prompt with grounding instructions |
| FastAPI application | Endpoints: POST /extract, POST /predict, POST /explain (or a combined POST /run) |
| Docker setup | Dockerfile + docker-compose.yml to run the full system |
| Test cases | At least 3 complete end-to-end test inputs with expected behavior documented |
